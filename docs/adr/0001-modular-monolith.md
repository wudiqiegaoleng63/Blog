# ADR-0001：采用模块化单体与 MySQL 队列优先

- 状态：已接受
- 日期：2026-07-13
- 决策范围：Stage 0 及后续业务模块的初始演进路径

## 背景

项目当前处于 Stage 0，需要先建立可靠、低运维成本的后端基础：统一配置、HTTP 服务、数据库迁移、健康检查、后台进程入口和容器化部署。团队尚未获得足以证明微服务拆分、独立消息集群或跨服务一致性成本合理的业务规模与流量数据。

同时，后续阶段预计会出现认证、文章、评论、后台任务以及 AI/RAG 等能力。它们需要清晰边界，但在早期也需要共享事务、统一迁移和快速迭代。

## 决策

### 1. 采用模块化单体

后端保持一个 Go module 和一个部署代码库，按领域模块、平台能力和共享约定划分内部边界。模块通过明确接口协作；避免跨模块直接修改内部状态，并避免把所有逻辑堆入 HTTP handler 或公共工具包。

Stage 0 仅有 operations（健康检查）模块。认证、文章、评论和 AI 等目录或配置即使预留，也不视为已实现。

### 2. API 与 Worker 使用同一镜像

`api`、`worker`、`migrate` 从同一源码和同一 Dockerfile 构建进一个 `blog-backend` 镜像，通过容器 command/entrypoint 选择进程角色：

```text
blog-backend + command api      -> HTTP API
blog-backend + command worker   -> background worker
blog-backend + command migrate  -> schema migration CLI
```

API 与 Worker 是独立进程，可独立扩缩容和重启，但共享版本、领域代码、配置模型和数据库 schema。Worker 在 Stage 0 只是生命周期占位，不代表任务消费已经完成。

### 3. 后台任务先使用 MySQL queue

早期后台任务优先基于 MySQL 的 `background_jobs` 表实现。生产业务状态和任务入队可在同一数据库事务中协调，先获得明确的领取、锁、重试、幂等和审计语义，再根据实际吞吐与延迟数据优化。

Stage 0 只创建了表和配置，尚未实现生产者或消费者。

### 4. Kafka 后置

当前不引入 Kafka。只有在观测数据证明 MySQL queue 无法满足吞吐、延迟、消费者隔离、重放或多系统事件分发需求时，才单独提出 ADR，引入 Kafka 或其他消息基础设施。

Kafka 后续若引入，不应直接替代业务事务中的可靠写入；需要先定义 outbox、事件 schema、兼容策略、幂等消费、保留期和故障恢复责任。

## 理由

- 降低 Stage 0 的部署、监控、备份和本地开发成本。
- 共享 SQL 事务，避免在业务模型尚不稳定时提前承担分布式一致性。
- 单镜像消除 API/Worker 版本漂移，复用领域规则和 migration 工具。
- 进程角色仍然分离，为资源隔离和独立扩缩容保留空间。
- 模块边界允许未来按证据拆分，而不是因早期基础设施选择锁死。

## 后果

### 正面

- 一个构建产物覆盖 API、Worker 和 migration，发布与回滚路径一致。
- 本地和 CI 只需 Go、MySQL、Redis 与 Compose，不需要消息集群。
- 业务写入与任务入队可设计为原子事务。
- SQL migration 是统一 schema 真相来源。
- 模块可在单进程内低成本调用，减少网络协议和序列化负担。

### 负面与约束

- 单镜像变更会使 API/Worker 一起发布，即使只修改一个角色。
- 模块边界依赖代码审查和测试执行，不由网络隔离强制保证。
- MySQL queue 会增加数据库负载；必须设计合适索引、短事务、锁超时、批量大小、退避与幂等。
- 长任务或高并发轮询可能与在线事务争用连接和 I/O。
- 模块化单体不能被误解成单进程：API 与 Worker 仍需分别运行和监控。
- 若未来拆分服务，需要提取稳定契约、数据所有权和可观测性，存在迁移成本。

## 触发重新评估的条件

出现以下一项或多项并有测量证据时，重新评估队列或服务边界：

- MySQL queue 的锁竞争或轮询负载影响在线请求 SLO。
- 任务吞吐、积压恢复时间或端到端延迟持续不达标。
- 多个独立消费者需要长期保留、重放或按分区有序的事件流。
- 不同模块需要独立数据所有权、发布节奏或合规隔离。
- API 与 Worker 的依赖或镜像体积显著分化，单镜像发布代价超过一致性收益。

重新评估必须基于指标和故障案例，并通过新的 ADR 记录迁移方案；本 ADR 不预先承诺 Kafka 的引入时间。
